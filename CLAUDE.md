## Жёсткие правила для UI работы

### Scope: только GitHub-модуль
- Все UI-правки касаются ТОЛЬКО файлов GitHub-модуля:
  - app/src/main/java/com/glassfiles/ui/screens/GitHub*.kt
  - app/src/main/java/com/glassfiles/ui/screens/components/ 
    (если там есть GitHub-специфичные компоненты)
- НЕ трогать общие компоненты приложения (BrowseScreen, FileManager, 
  AI Chat, Terminal, AppearanceScreen, SettingsScreen, и т.д.)
- НЕ трогать Theme.kt (исключение: явный запрос пользователя на 
  правку темы)
- НЕ трогать GitHubManager.kt — это data layer
- Если правка одного экрана GitHub-модуля требует изменения общего 
  компонента — остановись и спроси пользователя

### Никаких "украшательств"
Запрещено добавлять без явной просьбы:
- Градиенты на фоне карточек (никаких Brush.linearGradient, 
  Brush.verticalGradient, background = Brush.*)
- Серые/тёмные подложки под белыми/светлыми элементами
- Тени-стейны (shadow с большим radius который выглядит как пятно)
- Полупрозрачные слои с alpha поверх контента
- "Glassmorphism" эффекты с blur — только если уже есть в проекте 
  и применены везде одинаково

Если хочется добавить визуальный эффект для "глубины" — используй:
- Solid цвет из MaterialTheme.colorScheme.surface или surfaceVariant
- 1px hairline border через outlineVariant с alpha 0.06-0.12
- Минимальную elevation (1-2dp), не больше

### Цвета — только через MaterialTheme
- Никаких hardcoded hex (Color(0xFF4CAF50), Color.Green и т.д.)
- Все цвета через MaterialTheme.colorScheme.*:
  - Фоны: surface, surfaceVariant, surfaceContainer
  - Текст: onSurface, onSurfaceVariant
  - Границы: outline, outlineVariant
  - Акцент: primary (это то что юзер выбирает в Appearance)
  - Ошибки/успех: error, и для success — tertiary или явно 
    зелёный из палитры material
- Исключения (можно хардкод): семантические индикаторы статусов 
  GitHub (зелёный для success, красный для failed, жёлтый для pending) — 
  но и здесь предпочтительно использовать материал-токены если они есть

### Тестирование на обеих темах
Любая визуальная правка должна работать на:
- Light theme
- Dark theme
- AMOLED theme (это Dark с чёрным #000000 фоном)

Перед declaration "готово" — мысленно прокрути правку в каждой теме. 
Если не уверен как будет выглядеть — спроси пользователя или попроси 
скриншот после применения.

### Работа со скриншотами
- Если пользователь прислал скрин с пометками (стрелки, обводки) — 
  смотри внимательно ИМЕННО на пометки, это и есть проблема
- Не "улучшай" то что пользователь не просил улучшать
- Если на скрине баг — сначала пойми визуально что не так, потом 
  ищи в коде. Не наоборот.

### Принципы полировки UI

Когда пользователь говорит "сделай UI лучше" / "убери нейрослоп" — 
это не разрешение творить. Это просьба применить конкретные оси:

1. Типографика: hierarchy через размеры (24/18/14/12sp) и веса 
   (400/500/700), letter-spacing для caps, monospace для технических 
   данных (хеши, durations).
2. Spacing: ритм 4/8/12/16/24/32/48dp. Не ad-hoc числа.
3. Иконки: размеры 14dp/20dp/24dp по роли. Filled для primary, 
   Outlined для secondary.
4. Состояния: status через цвет + позицию + иконку, не только цвет.
5. Микроанимации: scale 0.98 на press, fade на появление. Без 
   агрессивных переходов.
6. Empty/loading states: skeleton-плейсхолдеры вместо спиннеров, 
   иконка+текст для empty.

### Если задача расплывчатая
Когда пользователь говорит "сделай красивее" / "улучши" — не 
действуй сразу. Спроси: "Что именно? Покажи скрин с проблемой / 
опиши что не нравится". Лучше потратить 30 секунд на уточнение чем 
сделать не то.

### Один экран за раз
Не делать массовые правки сразу по всему GitHub-модулю. Работать 
итерационно: один экран → пользователь смотрит → правки → следующий. 
Это снижает риск что какой-то общий компонент сломается во многих 
местах сразу.
